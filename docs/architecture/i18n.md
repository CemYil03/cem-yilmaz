# Internationalization (i18n)

## Context

The site serves German (default) and English visitors from one codebase. Without a deliberate locale strategy, routes, SEO (`hreflang`,
sitemap), and bilingual copy drift into ad-hoc patterns — or an i18n library is adopted before the catalog justifies it.

## Decision

**Optional path parameter `{-$locale}`** (TanStack Router) with **inline** `{ de: '…', en: '…' }[locale]` literals at the call site. No i18n
library; no page-level `COPY` const (see [conventions.md](../conventions.md#bilingual-copy)).

- Default locale is `de` (unprefixed URLs: `/`, `/about`). English is prefixed: `/en`, `/en/about`.
- Canonical `Locale` (`'de' | 'en'`), `LOCALES`, and `DEFAULT_LOCALE` live in `src/shared/locale.ts`. Web-only helpers
  (`localeFromAcceptLanguage`, `localeFromParam`) re-export those types from `src/web/utils/locale.ts`.
- `languageTagFromLocale(locale)` maps to BCP-47 tags used by `Intl`: `de` → `de-DE`, `en` → `en-US`.
- `src/routes/{-$locale}.tsx` `beforeLoad` auto-redirects first-time visitors from `Accept-Language` / `locale` cookie when the preferred
  locale is not the default.
- Components read locale via `useLocale()` (`src/web/hooks/useLocale.ts`). The language switcher (`LanguageSelector`) sets the cookie and
  navigates.
- **Display formatting** (dates, date ranges, month/year, currency) uses isomorphic helpers in `src/shared` — `formatDate`,
  `formatDateRange`, `formatMonthYear`, `formatCurrency`, plus storage helper `formatIsoDate`. Every display helper takes a required
  `{ locale }` options object and derives the language tag internally via `languageTagFromLocale`. Prefer these over local `Intl` /
  `date-fns` formatters in routes. Import from `src/shared` (barrel).
- Calendar widgets and relative time (`formatDistanceToNow*`) still use `DATE_FNS_LOCALE` from `src/web/utils/dateFnsLocale.ts` — `date-fns`
  needs its own locale objects and is not inferred from the route.

DB-backed bilingual content uses paired `*De` / `*En` columns — see [content-model.md](./content-model.md). SEO alternates come from
`seoMeta()` — see [discovery-seo.md](./discovery-seo.md).

## Alternatives Considered

| Option                                                  | Why rejected                                                                                                                                    |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| i18n library (Paraglide, react-intl)                    | Message catalogs and ICU are overkill for a two-locale site with mostly inline strings; Paraglide remains the upgrade path if the catalog grows |
| Router rewrite that strips locales from route awareness | Hides locale from route params; harder to type and to drive `seoMeta` / loaders                                                                 |
| Separate route trees per locale                         | Duplicates every page file                                                                                                                      |
| Colocated per-route `formatDate` / `formatCurrency`     | Drifted (`en-GB` vs `en-US`, null handling); unusable from server email/agent rendering                                                         |

## Consequences

- Every visitor-facing string is duplicated inline — cheap to read, easy to drift if a string is updated in only one locale.
- Adding a locale means extending `LOCALES` / `Locale` and touching every inline object (plus cspell dictionary if needed).
- Workspace chrome intentionally hides the language switcher (English-only admin surface); public pages keep it.
- Day-to-day recipes (new page under `{-$locale}/`, sitemap entry) live in [conventions.md](../conventions.md) and the "How to Add Things"
  section of [`AGENTS.md`](../../AGENTS.md) — this ADR stays on _why_.
- Server and client share one formatting surface under `src/shared`, so email HTML and agent snapshots stay consistent with the UI.

| ------------------------------------- | ------------------------------------------------- |
